# site-packages-test

Tests that REACHABLE excludes `site-packages` directories from CWE/SAST scanning.

## The Bug

A customer scanned a repo and got flagged for `xmlrpc` usage inside
`pip/_internal/commands/search.py` — code that belongs to pip, not to the
customer's application. The customer's code never uses xmlrpc.

## Root Cause

Semgrep was walking into `site-packages` and finding CWE patterns in
third-party library internals. This happened because:

1. The venv had a non-standard name (not `.venv` or `venv`)
2. `site-packages` was not in the exclude list

## Fix

Replaced `.venv` / `venv` exclusions with `site-packages` in `EXCLUDE_DIRS`.
This catches every venv naming convention — `.venv`, `venv`, `myenv`,
`virtualenv`, system Python, conda — because they all contain `site-packages`.

## Test Scenarios

| Directory | Old behavior | New behavior |
|---|---|---|
| `.venv/lib/.../site-packages/` | Excluded (`.venv` match) | Excluded (`site-packages` match) |
| `venv/lib/.../site-packages/` | Excluded (`venv` match) | Excluded (`site-packages` match) |
| `myenv/lib/.../site-packages/` | **SCANNED (bug)** | Excluded (`site-packages` match) |
| `conda_env/lib/.../site-packages/` | **SCANNED (bug)** | Excluded (`site-packages` match) |

## Expected Results

- `app.py` — has one intentional CWE finding (reachable SQL injection). MUST be detected.
- Everything under `*/site-packages/` — MUST NOT produce any findings.

## Validation

Run a scan, then check that:
1. The SQL injection in `app.py` IS found
2. No findings have a path containing `site-packages`
